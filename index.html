0安全审查(推送前必做)

目标:确认公开仓库不会泄露密钥。Public 仓库一旦推送,历史几乎无法彻底清除。

  • 检查 git 历史,确认 .env 从未被提交: 
    git log --all --full-history -- .env
git log --all --full-history -- "*.env"
  • 全文搜索历史中是否有硬编码密钥: 
    git log -p -S "JWT_SECRET" --all
git log -p -S "DATABASE_URL" --all
  • 若历史有泄露,用 git filter-repo 清理(危险操作,需二次确认)
  • 确认 .gitignore 已覆盖 .env / *.env / node_modules / dist / dev.db

1数据库迁移:SQLite → PostgreSQL

🔴

这是必做项,不是可选项。几乎所有云平台文件系统是临时的,SQLite 文件会在重启后丢失。

1.1 修改 Prisma provider

文件:apps/api/prisma/schema.prisma

datasource db {
  provider = "postgresql"   // sqlite → postgresql
  url      = env("DATABASE_URL")
}

1.2 更新环境变量示例

文件:.env.example

DATABASE_URL="postgresql://user:pass@host:5432/dbname?schema=public"
JWT_SECRET=change-me-to-a-strong-random-string
JWT_EXPIRES_IN=7d
PORT=3000
NODE_ENV=production
CORS_ORIGINS=https://your-app.onrender.com

1.3 删除旧迁移,重新生成

# 删除旧的 SQLite 迁移目录
rm -rf apps/api/prisma/migrations

# 用本地 Postgres(走 docker-compose)生成初始迁移
pnpm --filter api exec prisma migrate dev --name init

1.4 加生产迁移命令

文件:apps/api/package.json 的 scripts 追加:

"start": "prisma migrate deploy && node dist/app.js",
"db:migrate:deploy": "prisma migrate deploy"
⚠️

生产环境用 migrate deploy,不要用 migrate dev。后者会重置数据。

2生产代码适配

2.1 CORS 动态化

文件:apps/api/src/app.ts:22-25。当前写死 localhost,部署后前端域名不同会跨域。

const allowedOrigins = process.env.CORS_ORIGINS
  ? process.env.CORS_ORIGINS.split(',')
  : ['http://localhost:5173', 'http://localhost:3000'];

app.use(cors({
  origin: allowedOrigins,
  credentials: true,
}));

2.2 JWT_SECRET 生产强制校验

文件:apps/api/src/config/index.ts。绝不允许默认密钥上生产。

const jwtSecret = process.env.JWT_SECRET;
if (!jwtSecret && config.nodeEnv === 'production') {
  throw new Error('JWT_SECRET must be set in production');
}

jwt: {
  secret: jwtSecret || 'dev-only-insecure-secret',
  expiresIn: process.env.JWT_EXPIRES_IN || '7d',
}

2.3 静态文件路径健壮性

文件:apps/api/src/app.ts:44。启动时检查前端产物是否存在:

const webDist = path.join(__dirname, '../../web/dist');
if (fs.existsSync(webDist)) {
  app.use(express.static(webDist));
  app.get('*', (req, res) => {
    if (!req.path.startsWith('/api')) {
      res.sendFile(path.join(webDist, 'index.html'));
    }
  });
} else {
  logger.warn('web/dist not found, running API-only mode');
}

5GitHub Actions 自动构建 CI

新建文件:.github/workflows/ci.yml

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with:
          version: 8
      - uses: actions/setup-node@v4
        with:
          node-version: 18
          cache: 'pnpm'
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm --filter api exec prisma generate
      - run: pnpm build

效果:每次推送到 main 或提 PR,自动 install + lint + build,失败会阻止后续部署。

🎯交接说明

📋

建议执行顺序:阶段 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7

关键提醒

  1. 数据库迁移是必做项,SQLite 在云端不可用
  2. 前后端同源架构已就绪,部署时只需部署后端一个服务,前端由后端托管静态文件
  3. CORS 目前硬编码 localhost,部署前必须改成环境变量
  4. JWT_SECRET 不能用默认值上生产,必须强制校验
  5. 平台和数据库选择需先问用户,推荐:Render + Neon
  6. Git 历史安全审查(阶段 0)必须在推送前完成,尤其是 Public 仓库
